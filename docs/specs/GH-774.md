# GH-774: Split `session.yaml` — durable prefs vs ephemeral state, provisioned by the worktree hook

> SPDD spec (REASONS). Design authority: **GH-774 comment 3** (supervisor
> direction). This spec supersedes the ticket's original "add exact-path
> permission rule" premise and reconciles the sibling ticket #784. It is the
> single source of truth for the change; `Dev10x:spec-sync` checks the diff
> against it before merge.

## Objective

`.claude/Dev10x/session.yaml` conflates two concerns with different lifetimes:

- **Durable repo preferences** — `friction_level`, `active_modes` — should be
  consistent across every worktree of a repo and rarely change.
- **Ephemeral per-worktree state** — `branch`, `tickets`, continuation prompts
  — is specific to one worktree/session and must never leak between them.

Because both live in one gitignored file under `.claude/`, writing/creating it
trips Claude Code's **`.claude/` self-edit gate** (the "allow Claude to edit its
own settings" prompt), which no ordinary `Write(.claude/…)` allow-rule
satisfies. The result is a permission prompt on `afk` / `session-config-seed` /
work-on Phase 0 in worktree after worktree.

**The permission-rule track is a dead end** (GH-774 comment 1): new worktrees
*do* inherit the `Write(.claude/Dev10x/session.yaml)` allow rule, yet a fresh
session still prompts, because the write goes through the self-edit gate, not a
`Write()` allow-rule.

## Technical Approach

Resolved design (GH-774 comment 3): **keep both files under `.claude/Dev10x/`;
escape the self-edit gate by making Claude *read, not write*.** The worktree
`post-checkout` hook provisions both files so no runtime `Write(.claude/…)`
happens on the hot path and the gate never fires.

Split by lifetime into two files:

| File | Contents | Git | Worktree hook | Written by |
|------|----------|-----|---------------|-----------|
| `.claude/Dev10x/session.yaml` | **Ephemeral**: `branch`, `tickets`, `tasks`, `continuation_prompt`, `insights` | gitignored | **seeded fresh** (`dev10x session seed`), **excluded from copy** | hook at create; never carried between worktrees |
| `.claude/Dev10x/config.yaml` | **Durable**: `friction_level`, `active_modes`, `gate_preset`, `gate_overlays`, `gate_overrides` | gitignored (personal, like `.idea/`) | **copied** source→worktree by the existing `.claude/` rsync | user edits rarely; a rare write-prompt there is acceptable |

This satisfies the three requirements the supervisor named without git-tracking:
- **(a)** gate escaped → Claude reads both; the hook writes them.
- **(b)** shared across worktrees → durable prefs **copied** by the hook.
- **(c)** ephemeral never stale → `session.yaml` seeded per worktree, never copied.

## Architecture

### Python side — **centralized** (low risk)

`src/dev10x/domain/documents/session_yaml.py` — class `SessionYamlDocument`
owns the file path, all reads (`read_friction_level`, `read_active_modes`,
`read_friction_and_modes`, `read_gate_policy_inputs`, `read_session_identity`),
and the canonical `render()` / `write()`. Every `.py` consumer routes through it
(`session/service.py`, `domain/documents/session_context.py`, `hooks/session.py`,
`mcp/gate_tools.py`). Downstream policy/rule code (`domain/gate_policy.py`,
`domain/session_rules.py`) receives values as **parameters** — no file I/O, no
change needed.

**Change shape:** introduce a sibling `ConfigYamlDocument` (or extend
`SessionYamlDocument` with a `config_path` and durable-key accessors) so:
- durable keys read from `config.yaml`, with **fallback to `session.yaml`**
  during migration (a legacy `session.yaml` still carrying `friction_level`
  keeps working until re-provisioned);
- ephemeral keys read/write `session.yaml` only.

### CLI side

- `src/dev10x/commands/session.py` (`session seed`) currently seeds the
  **durable** keys (`friction_level`, `active_modes: []`). **Flip it** to seed
  ephemeral-only, and add a path (new subcommand or hook step) that provisions
  `config.yaml` with durable defaults when absent.
- `src/dev10x/commands/init.py` (`_seed_project`, `init`) writes via
  `SessionYamlDocument.render()` — update to the split shape.

### Worktree provisioning — **the risky part** ⚠️

Templates: `skills/git-worktree/templates/post-checkout-{python-uv,node,node-husky}.sh`.
They `copy_clean ".claude/" worktrees` then run the GH-705 `dev10x session seed`
block. Required change: **per-file handling** — copy `config.yaml`
source→worktree, but **exclude + seed** `session.yaml` (so a copied stale
`branch:` never leaks).

> **Root cause — CONFIRMED by static trace of the shipped template**
> (`skills/git-worktree/templates/post-checkout-python-uv.sh:51-80`). The bug is
> real and lives in `copy_folder`'s dirty-exclusion logic:
>
> - `copy_clean ".claude/" worktrees` → `copy_folder` builds its rsync excludes
>   from `git status --porcelain` of the source repo (line 56:
>   `grep "^.claude/" "$DIRTY_LIST" | sed "s|^.claude/||" > dir_excl`), then
>   `rsync -a --exclude-from="$dir_excl"`.
> - In a repo that gitignores **only** `session.yaml` narrowly (tt-pos), the
>   sibling `.claude/Dev10x/auto-advance-records.md` is untracked-and-unignored,
>   so `git status --porcelain` **collapses** it to a single `?? .claude/Dev10x/`
>   entry. `sed` strips `.claude/` → the exclude pattern becomes `Dev10x/` →
>   rsync **drops the entire `.claude/Dev10x/` directory** from the worktree copy.
>   This is exactly comment 1's mechanism.
> - This repo (Dev10x-Claude) gitignores `.claude/Dev10x/` **wholesale**, so
>   nothing under it appears in `--porcelain`; the dir is NOT excluded and copies
>   via plain `rsync -a` — carrying a **stale `branch:`** (the other half of the
>   problem). This is why the bug shows in tt-pos but not here.
>
> **Consequence for the fix:** you can neither blanket-copy (`session.yaml`
> carries a stale `branch:`) nor blanket-exclude (`config.yaml` is dropped)
> `.claude/Dev10x/`. **Per-file handling is mandatory.** Concretely: (i) run the
> source `git status --porcelain` with `-uall` so untracked files list
> individually instead of collapsing the dir; and/or gitignore
> `auto-advance-records.md` so it never triggers the collapse; **and** (ii) add
> an explicit `--exclude=Dev10x/session.yaml` to the `.claude/` copy so the
> ephemeral file is never carried, keeping the GH-705 seed block to regenerate
> it. `config.yaml` (durable) is then copied because it is gitignored → absent
> from `--porcelain` → not excluded.

### Precedent

precedent — but it is git-tracked (a shared *project norm*). The new
`config.yaml` differs deliberately: **personal, gitignored, hook-copied.**

     exclusion mechanism (per-file, not blanket dir exclude).

   writes (ephemeral→`session.yaml`):

## Technical Risks

**Risk: hook fix targets the wrong layer** *(primary)*
- Scenario: implement a template exclude that doesn't match the actual live bug
  (see [Verify]) → prompt persists, or a legit copy is dropped.
- Mitigation: reproduce in a fresh Dev10x worktree first; confirm layer before edit.

**Risk: migration drops durable prefs**
- Scenario: existing `session.yaml` carries `friction_level`; after the split a
  reader looks only at `config.yaml` and silently defaults to `guided`.
- Mitigation: durable readers fall back to `session.yaml` until re-provisioned;
  add a one-time migration that lifts durable keys into `config.yaml`.

**Risk: scattered skill-doc drift**
- Scenario: one skill still writes durable keys into `session.yaml`, re-coupling.
- Mitigation: grep-gate in review; `spec-sync` structural check.

---

## Rollout Considerations

- **Migration:** on first read after upgrade, if `config.yaml` is absent but
  `session.yaml` carries durable keys, lift them into `config.yaml` (or fall
  back transparently). No data loss; idempotent.
- **Deploy:** plugin release; existing sessions keep working via fallback.
- **Rollback:** revert; both files stay gitignored, no tracked-state change.

## Safeguards

**Invariants (must hold post-change):**
- No runtime `Write` to `.claude/Dev10x/session.yaml` or `config.yaml` on the
  work-on hot path in a properly provisioned worktree (the hook writes; Claude
  reads) — the self-edit gate never fires.
- A worktree's `session.yaml` never contains another worktree's `branch:`.
- Durable prefs are identical across a repo's worktrees (copied, not drifting).

**Validation rules:**
- Durable readers tolerate a missing `config.yaml` (fall back to `session.yaml`,
  then to documented defaults) — never raise, never silently mis-default.
- `session seed` is idempotent (`O_EXCL`) — never clobbers an existing file.

**Authorization safeguards:**
- The split must not require broadening any allow-rule or touching the self-edit
  gate setting; escape is via provisioning, not permission escalation.
