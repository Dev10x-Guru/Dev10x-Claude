## Cross-Phase Convergences

The same root cause surfaced in multiple phases — these are the
strongest investment signals.

### 1. Untyped Task / dict-of-Any propagation (B1, B10)

`Plan.tasks: list[dict[str, Any]]` flows through `Plan`,
`PlanSummary`, `DecisionGuidanceRule`, and `task_plan_sync` with
~15 string-key call sites. A `Task` value object is the
prerequisite for type-safe plan refactors elsewhere.

### 2. AllowRule parsing / matching / building duplicated 4–5× (C2, C3, C9)

Four independent implementations of `Tool(pattern)` matching
(`hooks/permission_diagnostics.py`, `validators/prefix_friction.py`,
`skills/audit/analyze_permissions.py`,
`skills/permission_investigator/report.py`) have already drifted
semantically. Five settings-loader copies, four rule-string
builders. This is the single largest correctness risk in the
audit — contradictory diagnostics from the same allow rule.

### 3. Hook ↔ audit ↔ skills boundary inversion (D1, I1, I2, I6, I10)

- `domain/session_document.py` lazy-imports `hooks.task_plan_sync`
  (domain → hooks)
- `audit/analyze.py` imports `skills.audit.analyze_permissions`
  internals (audit → skills)
- `hooks/audit_emit.py` imports `audit.log_reader` internals
  (hooks → audit)
- `session/queries.py` defers imports of `hooks.session_policy`
  to paper over a cycle

Cumulative chain: `hooks → audit → skills`. No stable protocol
demarcates context boundaries.

### 4. Non-atomic writes alongside an atomic-write helper (E2, E3, E8)

`atomic_write_text` and `file_lock` exist and are used correctly
in `task_plan_sync.cmd_hook` and `Plan.save()`. They are
**bypassed** by:
- `domain/session_document.write_state` (bare `path.write_text`)
- `plan.service.set_plan_context` / `archive_plan` (no lock)
- audit JSONL appends (relies on TextIOWrapper buffered atomicity)

Lock-sidecar naming is also fragmented: `locked_json_update`
strips suffix, `file_lock` appends `.lock` (E10 — documented but
unenforced).

### 5. MCP boundary `Result[T].to_dict()` contract violations (A4, B5, H1, I9)

- `SuccessResult.to_dict()` has a polymorphic `isinstance(value,
  dict)` branch that silently changes output shape
- `record_upgrade` MCP handler returns raw dict, bypassing
  `Result[T].to_dict()` (single outlier across ~40 handlers)
- No `ResultProtocol(Protocol)` to make the contract type-checked

### 6. GH-979 effective-CWD discipline gaps (H2, H6, H11)

`subprocess_utils.effective_cwd()` is the documented contract;
three modules call `os.getcwd()` directly (`domain/events/
hook_input.py:27`, `hooks/permission_diagnostics.py:202`,
`skills/audit/analyze_permissions.py:254`). Seven skills modules
call `subprocess.run` directly. Two hook modules cache
`GitContext()` at module scope despite its docstring warning.

### 7. Behavioral dispatch via `if/elif` instead of registries (A9, F1, F2, F4, F9)

Three parallel "rule" abstractions (`Rule` data object,
`*Rule` policy classes, `Validator` chain objects). Tool-name
classification chains (`analyze_actions.py`), kind dispatchers
(`github_app.py`), and friction-level branches (`session_policy.py`,
`skill_redirect.py`) all reinvent dispatch without using the
declarative `TASK_TRANSITIONS` pattern already in `plan.py`.

### 8. Batch `gh` API built but never wired (G4, I3, I4)

`github/queries.py::batch_find_prs` shipped in PR#203 with 15
unit tests; zero callers. `collect_prs.py` still N+1s over
tickets. PR#203 deferred wiring as GH-146.

### 9. Skills-as-uv-scripts duplicate infrastructure (G2, H3, H7, I3)

`skills/release/collect_prs.py` (250 LOC), `skills/monitor/
pr_notify.py`, `skills/monitor/ci_check_status.py` cannot import
from `dev10x` because they ship as standalone uv-scripts with
`dependencies = []`. Result: duplicated `extract_jtbd`,
`md_to_slack_bold`, `load_config`, ticket-ID regex, and the
batch API stays disconnected.

### 10. Test coverage gaps on recent shipping skills (G1, G3, G7)

- 4 MCP tool handlers shipped via GH-205/220
  (`pr_issue_comment`, `minimize_comments`, `request_review`,
  `resolve_review_thread`) have zero tests
- `Dev10x:fanout`, `Dev10x:work-on` (most-traffic supervisor
  skills) have zero Python tests
- `dev10x playbook diff` CLI has no `CliRunner` test

## ADR Candidates

HIGH-impact items that imply architectural decisions and would
benefit from ADRs before implementation:

- **ADR-0007 Rule/Policy archetype unification** — A9 + D3 +
  F4 — currently three incompatible rule mechanisms
- **ADR-0008 Context boundary protocol** — I1, I2, I6 chain —
  hook/audit/skills coupling needs an explicit dependency rule
- **ADR-0009 Result[T] contract at MCP boundary** — A4 + B5 +
  H1 — uniform shape + protocol enforcement
- **ADR-0010 uv-script skills vs importable modules** — G2 +
  I3 — `dependencies = []` constraint is forcing duplication
- **ADR-0011 Atomicity model for shared state** — E2 + E3 +
  E8 + A12 — codify the `atomic_write_text` + `file_lock` rule
  and document the lock-sidecar convention

## Notes

- Phase F surfaced zero HIGH-impact findings; all behavioral
  patterns are MEDIUM polish.
- Phases G and B both flagged the Task VO cascade and SKILL.md-
  only orchestration as the riskiest gaps shipping today.
- Phase E's three HIGH findings (E2, E3, E8) are the cheapest
  wins in the entire audit — S effort, immediate correctness.
- Two areas were audited and found **compliant**: pathlib
  usage and `yaml.safe_load` discipline. The LazyGroup CLI
  pattern is compliant today but lacks regression protection.
